# CGU v2: 核心創意機制

> **「不是模擬創意，而是實現創意的結構」**

---

## 🎯 核心洞察

### 為什麼 v1 是假創意？

```
v1: 方法論模板 + Prompt + 隨機碰撞 = 創意的模擬
v2: 結構同構 + 路徑探索 + 對抗進化 = 創意的機制
```

### 真正的創意是什麼？

Arthur Koestler 的 **Bisociation**：
> 創意 = 兩個原本不相關的「思維矩陣」突然找到**結構上的連結點**

關鍵詞：**結構同構**、**非顯而易見**、**意外但有意義**

## 🔬 Engine 1: AnalogyEngine（類比搜尋器）

### 核心理念

類比不是「找相似的詞」，是「找結構相同的問題」。

```
問題結構抽象：

「軟體技術債」
    ├─ 結構：累積的隱藏問題
    ├─ 動態：延遲處理 → 代價增長
    ├─ 解法模式：定期清理 vs 重構 vs 忽視
    └─ 利害關係：短期省力 ↔ 長期風險

跨域搜尋相同結構：

「建築結構補強」     「身體健康管理」     「生態環境債」
    ├─ 累積的結構問題   ├─ 累積的健康問題    ├─ 累積的環境問題
    ├─ 延遲修復 → 危險  ├─ 延遲治療 → 惡化   ├─ 延遲治理 → 災難
    ├─ 補強 vs 重建     ├─ 預防 vs 治療      ├─ 保護 vs 修復
    └─ 成本 ↔ 安全      └─ 方便 ↔ 健康       └─ 發展 ↔ 永續
```

### 類比品質評估

```python
analogy_quality = {
    "structural_match": 0.8,    # 結構匹配度
    "surface_distance": 0.7,    # 表面差異（越大越好）
    "insight_potential": 0.9,   # 洞察潛力
    "transferability": 0.6,     # 可遷移性
}
```

## 🕸️ Engine 2: ConceptGraph（概念圖譜遍歷）

### 核心理念

在知識空間中，創意 = 找到「非顯而易見但有意義」的路徑。

```
直接路徑（無聊）：
咖啡 ──drink──► 飲料 ──stimulant──► 提神

間接路徑（有創意）：
咖啡 ──origin──► 衣索比亞 ──trade──► 殖民歷史 
     ──globalization──► 供應鏈 ──outsourcing──► 程式外包
     ──coding──► 程式設計

發現：咖啡和程式設計的深層連結
- 都是全球化的產物
- 都有「產地」vs「消費地」的不對稱
- 都涉及「提取價值」的過程
```

### 路徑評估

```python
path_quality = {
    "hop_count": 5,              # 跳數（3-7 最佳）
    "semantic_coherence": 0.7,   # 語義連貫性
    "novelty_score": 0.8,        # 新穎度（非常規路徑）
    "explainability": 0.9,       # 可解釋性（能說得通）
}
```

## ⚔️ Engine 3: AdversarialEngine（對抗式創意）

### 核心理念

真正的創新來自「被迫突破」，不是「隨便想想」。

```
Round 1:
  Generator: 「用 AI 自動寫程式碼」
  Critic: 「GitHub Copilot 已經做了，不夠新」
  
Round 2:
  Generator: 「AI 自動修復 bug」
  Critic: 「也有了，而且效果不好」
  
Round 3:
  Generator: 「AI 預測哪段程式碼會出 bug，在寫之前就警告」
  Critic: 「有點意思，但如何預測？」
  
Round 4:
  Generator: 「根據開發者的疲勞度、時間、程式碼複雜度，預測 bug 風險」
  Critic: 「這個角度比較新，可以深入」
  
最終：從「AI 寫程式」進化到「AI 預測人類錯誤風險」
```

### 對抗規則

```python
adversarial_rules = {
    "no_retreat": True,        # 不能換題目
    "must_address": True,      # 必須回應攻擊
    "escalating_difficulty": True,  # 攻擊越來越難
    "max_rounds": 5,
}
```

## 🔄 三引擎協作流程

```
輸入：「如何讓遠端工作更有效率？」

Step 1: ConceptGraph 展開問題空間
────────────────────────────────
    遠端工作
    ├── 溝通 ─── 非同步 ─── 時區 ─── 全球化
    ├── 信任 ─── 監控 ─── 自主 ─── 責任
    ├── 工具 ─── 視訊 ─── 協作 ─── 虛擬
    └── 文化 ─── 孤獨 ─── 歸屬 ─── 儀式

Step 2: AnalogyEngine 跨域搜尋
────────────────────────────────
    結構：「分散式協作 + 信任建立 + 文化維繫」
    
    類比 1: 遠洋航海（分散的船隊如何協作）
    類比 2: 游牧民族（移動中的社群如何維繫）
    類比 3: 開源社群（純線上如何建立信任）
    類比 4: 修道院（分散但有共同儀式）

Step 3: AdversarialEngine 對抗進化
────────────────────────────────
    初始想法：「從游牧民族學習 - 定期聚會」
    
    攻擊 1: 「太貴了，而且疫情不適合」
    進化 1: 「虛擬聚會 + 實體小聚」
    
    攻擊 2: 「虛擬聚會沒有效果，大家都在滑手機」
    進化 2: 「有明確儀式的虛擬聚會 - 像修道院的晨禱」
    
    攻擊 3: 「什麼儀式？太宗教了」
    進化 3: 「每日 15 分鐘非工作話題分享 - 咖啡時間儀式」

最終輸出：
────────────────────────────────
「遠端工作的修道院模式」
- 每日固定時間的「非工作」視訊（咖啡儀式）
- 季度實體小聚（游牧式聚會）
- 異步工作但同步儀式
- 來源類比：修道院 × 游牧民族 × 開源社群
```

## 📊 創意品質評估框架

### Novelty-Usefulness-Surprise (NUS) 模型

```
                Novelty（新穎度）
                    ▲
                    │
                  0.8 ┼───────●───────┐
                    │       ╱        │
                    │      ╱         │
                  0.5 ┼─────╱──────────┤
                    │    ╱           │
                    │   ╱            │
                  0.2 ┼──╱─────────────┤
                    │ ╱              │
                    └─┴──────┴───────┴──► Usefulness（實用性）
                     0.2    0.5    0.8
                     
    圓圈大小 = Surprise（驚喜度）
    
    目標區域：高新穎 × 高實用 × 高驚喜
```

### 評估維度

| 維度 | 定義 | 測量方式 |
|------|------|----------|
| **Novelty** | 與已知方案的距離 | Embedding 距離 + 專利/論文查重 |
| **Usefulness** | 解決問題的程度 | 約束滿足度 + 可行性評估 |
| **Surprise** | 「啊哈！」的程度 | 路徑非顯而易見度 × 結構匹配度 |
